1. Executive Summary
MedVault DeFi is a blockchain-powered medical data platform built on the Hedera Hashgraph network. It creates a trusted ecosystem where patients securely store and share their medical records, family history, and diagnostic results — and where verified doctors can authenticate that data using Hedera-based micro-transactions. Authenticated records are then available for purchase within a decentralized marketplace, enabling healthcare research, insurance underwriting, and clinical decision support while preserving patient privacy and consent.

Core Value Proposition
Patients control their medical data. Doctors earn HBAR for verified attestations. Researchers and insurers access high-integrity health datasets — all governed by Hedera smart contracts with full auditability.

2. Problem Statement
The global healthcare system suffers from three critical data challenges:

Medical records are siloed across hospitals, clinics, and labs with no portable, patient-owned format.
There is no trusted, decentralized mechanism for a doctor to attest to the authenticity of a patient's records.
High-quality, consented medical datasets are scarce for researchers and insurers, limiting innovation.

MedVault DeFi addresses all three gaps by combining decentralised identity (DID), Hedera Token Service (HTS), and smart contract logic into a seamless, consumer-grade web application.

4. Core Features
4.1 Patient Account & Data Upload
Self-sovereign identity registration using Hedera DID standard.
Upload medical documents: PDF reports, PNG scan images, lab results, family history files.
Files encrypted client-side (AES-256) before storage on IPFS; only the IPFS CID hash is anchored on Hedera via HCS (Hedera Consensus Service).
Patient dashboard: view upload history, verification status, consent toggles, and HBAR earnings.
Consent management: grant or revoke data access per doctor or per buyer at any time.

4.2 Doctor Account & Credentialing
Doctors register with a medical licence number, specialty, and jurisdiction.
Platform admin reviews and approves credentials off-chain; approval mints a soulbound NFT (non-transferable) to the doctor's Hedera wallet as a verifiable credential.
Doctor dashboard: view pending verification requests, earnings ledger, and completed attestations.
Each doctor has a publicly visible Trust Score (0–100) computed from: volume of verifications, dispute rate, and peer ratings.

6. Detailed User Flows
6.1 Patient Registration & Upload Flow
Patient visits MedVault DeFi and clicks Create Account.
Patient connects a Hedera-compatible wallet (HashPack, Blade) or creates a custodial wallet via email.
A Hedera DID is auto-generated and associated with the wallet address.
Patient completes basic profile: name, date of birth, country (stored off-chain, hashed on-chain).
Patient uploads medical file (PDF or PNG). The app encrypts the file client-side and uploads to IPFS.
The IPFS CID and document metadata are published to a private HCS topic owned by the patient.
Patient selects a doctor from the marketplace and requests verification, locking the fee in escrow.

6.2 Doctor Registration Flow
Doctor visits MedVault DeFi and clicks Create Doctor Account.
Doctor submits: full name, medical licence number, issuing country, specialty, and wallet address.
Platform admin verifies credentials against national medical licence registries (manual or API-based).
Upon approval, platform mints a soulbound NFT to the doctor's wallet via HTS — this is their on-chain credential badge.
Doctor sets their verification fee (within platform limits) and specialty tags.
Doctor profile is published to the marketplace, visible to patients searching for verifiers.

7.2 Doctor as Data Buyer — In-Platform Read Access (MVP)
Doctors are first-class buyers in the MedVault marketplace. Unlike external researchers who download raw files, doctors interact with a secure in-browser viewer — they can read, annotate, and save records to their account without ever downloading the underlying file. Every access event is fully auditable on Hedera.

Key Design Principle: Read Without Download
When a doctor purchases a patient record, the platform decrypts the file in a sandboxed iframe viewer inside the doctor dashboard. The raw file bytes are never exposed to the browser file system. The doctor sees the document; they never hold the document.

7.2.1 Doctor Data Purchase Flow (MVP)
Doctor browses the Marketplace, filtered by specialty, document type, or condition tag.
Doctor views the anonymised listing: document type, patient age range, condition category, verification status, and price in HBAR. No PII is shown at this stage.
Doctor clicks Purchase Record. The DataMarketplace smart contract is triggered, locking the HBAR payment from the doctor's connected wallet.
Smart contract executes an atomic HBAR transfer: 92% routed directly to the patient's Hedera wallet, 8% to the platform treasury — all in a single transaction.
Upon confirmed payment, the patient's encrypted file decryption key is shared with the doctor's wallet via a one-time ECDH key exchange — scoped only to this purchase.
The file is fetched from IPFS and decrypted in the doctor's sandboxed in-browser viewer (iframe with Content-Security-Policy: no download, no clipboard). No file is written to disk.
The purchase is added to the doctor's Purchased Records library in their dashboard. They can re-open the in-browser viewer any time using their wallet key — no re-payment required.
An HCS message is published to the patient's topic confirming the sale: buyer DID, timestamp, transaction ID, and exact HBAR amount received.

7.2.3 Instant HBAR Transfer to Patient Wallet
Payment to the patient is not batched, delayed, or held in escrow — it is atomic and immediate. The DataMarketplace.sol contract executes the revenue split in a single Hedera contract call:

Smart Contract Logic — DataMarketplace.sol
function purchaseRecord(address patientWallet, uint256 price) external payable {
    require(msg.value == price, 'Incorrect payment');
    uint256 platformFee = (price * 8) / 100;
    uint256 patientShare = price - platformFee;
    payable(patientWallet).transfer(patientShare);    // instant to patient
    payable(platformTreasury).transfer(platformFee);  // platform cut
    emit RecordPurchased(msg.sender, patientWallet, price, patientShare);
}

The patient wallet receives HBAR within 3-5 seconds of the doctor confirming purchase — Hedera deterministic finality guarantees this.
The patient dashboard shows a real-time HBAR earnings notification with buyer specialty (not identity) and transaction amount.
The patient can withdraw or spend their HBAR balance at any time — no lock-up or vesting period applies.

8. Security & Privacy Model
All medical files are encrypted AES-256-GCM before leaving the patient's browser — MedVault never sees raw medical data.
Encryption keys are stored in the patient's wallet, never on MedVault servers.
Doctor access is temporary: the patient shares a one-time decryption key scoped to the verification request.
All access events are logged as HCS messages for full auditability.
GDPR & HIPAA design principles followed: data minimisation, purpose limitation, and right to deletion (patient can delete IPFS pin and revoke all consents).
Smart contracts are open-source and will undergo a third-party security audit before mainnet launch.

Privacy by Design
No raw PII is ever stored on-chain. Hedera anchors only hashes, CIDs, and attestation proofs. The platform cannot access, read, or sell patient data without an explicit, patient-signed smart contract consent transaction.
